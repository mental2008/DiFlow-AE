# Artifact Evaluation for DiFlow

## Opening Remarks
This guide provides step-by-step instructions for reproducing the main experiments and results in the DiFlow paper. The full evaluation is designed to complete in about **one hour**.

To simplify reproducibility, we provide a ready-to-use Docker image, `smarterlsy/diflow-ae:latest`, with all dependencies and configurations needed to run the experiments. This avoids complex environment setup. You can pull the image from [Docker Hub](https://hub.docker.com/repository/docker/smarterlsy/diflow-ae/general) and use it as described below.

To build the container yourself, see the `Dockerfile` in this repository.

## Dependencies and Requirements

### Hardware 
The artifact requires a machine with at least 8 NVIDIA GPUs. The system must support NVSHMEM and be properly configured with the required GPU interconnect, drivers, and software dependencies.

### Software
Detailed software dependencies are listed in `Dockerfile` and `requirements.txt`. Required models can be downloaded using `download_models.sh`.

To streamline the artifact evaluation process, we provide on-demand access to a preconfigured AWS instance that satisfies these hardware and software requirements.

## Run DiFlow in a Container

### Running on a Pod

If you are using the provided RunPod GPU Cloud environment, the pod has already been configured and the required Hugging Face models have already been downloaded. To download the models manually, run `download_models.sh`. We will provide the SSH command needed to log in to the pod.

After entering the pod, use the following commands to prepare for evaluation:

```bash
$ cd /workspace/DiFlow-AE

# Log in to your Hugging Face account to download models.
# You can skip this step when using the provided RunPod environment.
$ hf auth login

# Download the required models.
# You can skip this step when using the provided RunPod environment.
$ ./download_models.sh

# Generate worker_hostfile. This step is required.
# You can skip this step when using the provided RunPod environment.
$ ./ae-scripts/gen_hostfile.sh
```

### Running on a Server or VM

Use the following commands if you are running the artifact on a server or virtual machine.

```bash
# On your own machine, pull the prebuilt Docker image:
$ docker pull smarterlsy/diflow-ae:latest

# Stop any previously running container with the same name, if one exists.
$ docker kill diflow-ae || true

# Run the container in detached mode with GPU support.
$ docker run --gpus all --rm -it -d --name diflow-ae --shm-size 16G -v ./models:/root/models smarterlsy/diflow-ae:latest sleep infinity

# Access the container shell.
$ docker exec -it diflow-ae zsh

# Log in to your Hugging Face account to download models.
$ hf auth login

# Download the required models.
$ ./download_models.sh

# Generate worker_hostfile. This step is required.
$ ./ae-scripts/gen_hostfile.sh
```

## Major Claims
- **Correctness of Image Generation:** DiFlow does not alter the computation performed during diffusion inference and can generate images identical to those generated by Diffusers.
- **Speedup from Intra-/Inter-Node Parallelism:** DiFlow's intra-/inter-node parallelism achieves speedups comparable to existing implementations.
- **Effectiveness of Admission Control:** DiFlow's admission control improves SLO attainment under high request traffic.
- **End-to-End Performance Advantage:** In end-to-end evaluation, DiFlow outperforms monolithic diffusion serving systems.


## Reproduce the Main Experiments and Results

We first run three microbenchmarks to validate image-generation correctness, speedup from intra-/inter-node parallelism, and the effectiveness of admission control. We then run the end-to-end evaluation.

### Compare Images Generated by Diffusers and DiFlow (~5 minutes)
First, we validate the correctness of image generation with DiFlow. In Section 7.1 of the paper, we claim that DiFlow does not alter the computation performed during diffusion inference.

```bash
$ cd /workspace/DiFlow-AE

# First, run the Diffusers script to generate an image using Flux-Schnell.
$ python3 diffusers_inference/flux_schnell.py

# Second, generate an image with DiFlow.
$ ./ae-scripts/run_flux_schnell.sh

# Third, check whether the images are identical using cmp. It should return nothing.
$ cmp ./ae-results/verify_correctness/diflow_flux_schnell.png ./ae-results/verify_correctness/diffusers_flux_schnell.png
```

The `cmp` command is silent on success; it only prints output when files differ. If `cmp` returns no output, the two files are byte-for-byte identical.

The images generated by Diffusers and DiFlow should appear as follows:

Prompt | Model | Diffusers | DiFlow |
| --- | --- | --- | --- |
A cat holding a sign that says hello world | FLUX.1-schnell | <img src="ae-results/verify_correctness/diffusers_flux_schnell.png" width="200">  | <img src="ae-results/verify_correctness/diflow_flux_schnell.png" width="200"> |


Finally, you can inspect the model implementation and workflow composition code and compare it with Figure 6 and Figure 7 to review DiFlow's programming interfaces. Note that the figures show only the core workflow-composition logic.

```bash
# Validate Figure 6-bottom with the following command. Note that the load() function in Figure 6 is named initialize() in the code.
$ cat ./diffusionflow/operators/models/diffusion_models/flux_1_schnell.py

# Validate Figure 7 with the following command.
$ cat ./examples/register_flux_schnell_txt2img_workflow.py
```


### Speedup from Intra-/Inter-Node Parallelism (~10 minutes)

In Section 7.3 and Figure 10-left, we report the speedup achieved by *intra-/inter-node parallelism* in DiFlow.

#### Flux1-Schnell

Run the following scripts to measure the execution latency of intra-node parallelism (latent parallelism and CFG parallelism) and inter-node parallelism (ControlNet parallelism) with DiFlow. In the one-GPU setting, parallelism is disabled. In the two-GPU setting, parallelism is enabled.

```bash
# For the one-GPU setting, run:
$ ./ae-scripts/run_flux_schnell_one_gpu.sh

# For the two-GPU setting, run:
$ ./ae-scripts/run_flux_schnell_two_gpu.sh
```

Run the following script to analyze the results. The normalized latency will be printed to the console, and you can compare it with Figure 10-left.

```bash
$ python3 ae-scripts/print_normalized_latency.py
```

### Effectiveness of Admission Control (~10 minutes)

In Section 7.3 and Figure 10-right, we report the effectiveness of our admission control mechanism. This experiment reproduces the **S3** (Table 1) results in Figure 10-right.

```bash
# Run the following command to reproduce the results.
# It includes profiling, warmup, and the main experiments.
$ ./ae-scripts/eval_admission_control.sh
```
The SLO attainment with and without admission control will be printed to the console.

### End-to-End Evaluation (~40 minutes)

This experiment reproduces the results in Figure 9-(d), comparing the baselines and DiFlow in end-to-end evaluation.

```bash
# Run the following commands to reproduce the results.
# They include profiling, warmup, and the main experiments.
$ ./ae-scripts/run_baselines.sh
$ ./ae-scripts/run_diflow.sh

# Print the results to the console and generate the plot.
$ python3 ae-scripts/plot_end2end.py
```
The script prints SLO attainment under varying traffic-rate scales. The resulting figure is saved to `./ae-results/e2e_results.pdf`.
